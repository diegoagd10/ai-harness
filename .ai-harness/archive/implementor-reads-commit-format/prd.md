# PRD — implementor-reads-commit-format

- **Status**: draft
- **Date**: 2026-07-02
- **Change**: `implementor-reads-commit-format`
- **Author**: change-propose (phase 2 — PRD)

## Context

The per-task commit format the `change-implementor` agent writes is
single-sourced today: a single literal line in
`CODING_STANDARDS.md:63` — `` `[{change_name}][{task_id}] {slug}` ``.
The implementor prompt (`src/ai_harness/resources/change-agent/change-implementor.md`,
loop step 6, lines 99–100) only says "Include the task id and Change name
in the message" — it does **not** quote a template or reference
`CODING_STANDARDS.md`. The `## Commits` section is read at run time by
humans following `README.md`, not by the implementor itself.

The README already claims the format is "owned by
`CODING_STANDARDS.md ## Commits`" (`README.md:309-323`). The validator
mirrors that pattern for quality gates by reading the standards file
literally (`change-validator.md:262-265`). The implementor does not.
That gap means editing the standards file today does **not** change
what the implementor actually writes — only what the human thinks it
writes.

### Target users

- **OpenCode / Claude agents (`change-implementor`)** — must read and
  apply the format at every task commit.
- **The human owner of `CODING_STANDARDS.md`** — must be able to edit
  the format in one file and have the change propagate to every future
  per-task commit, with no agent-prompt edits.
- **`change-validator` (later, follow-up)** — out of scope here, but
  the design leaves room for it to gain a `git log` format check
  without re-shuffling the source-of-truth boundary.

### Why now

Three converging pressures:

1. The README already advertises a single source of truth that the
   implementor does not honor.
2. The validator already established the "read the standards file
   literally" pattern for gates. Mirroring it for commits removes the
   inconsistency.
3. As soon as someone edits `CODING_STANDARDS.md ## Commits` (e.g.
   switch to Conventional Commits), the implementor will silently keep
   writing the old format. Drift is undetectable today (validator
   does not run `git log`).

## Goals

- **G1.** When `change-orchestrator` delegates a task to
  `change-implementor`, it resolves the commit format by reading
  `CODING_STANDARDS.md ## Commits` and inlines the format string into
  the delegation.
- **G2.** The implementor uses that injected format literally at task
  commit time, substituting `{change_name}`, `{task_id}`, and `{slug}`
  before invoking `git commit -m`.
- **G3.** Editing `CODING_STANDARDS.md ## Commits` propagates to every
  subsequent per-task commit **without editing any agent prompt**.
- **G4.** The slug definition stays inside the implementor (no new
  scope). `{slug}` is generated by the implementor from the task
  title; the injected format is the contract for the rest of the
  message.
- **G5.** The implementor fails loud (`status: blocked`) if the
  orchestrator did not inject a usable format string (covers missing
  file, missing `## Commits` heading, or empty body).

## Non-goals (out of scope for this Change)

- **`change-archiver` `docs: archive {change}` format.** Separate
  concern. The archiver's commit is a scoped sentinel, not a
  per-task commit. Remains hardcoded in
  `change-archiver.md:79-81` and echoed in
  `change-orchestrator.md:641-642`.
- **`change-validator` `git log` enforcement.** The validator
  explicitly forbids `git log`, `git rev-parse`, `git cat-file`,
  `git diff`, `git show` (`change-validator.md:167-168`) and parses
  `implementation.md ## Commits` textually. A separate follow-up
  Change would be needed to add a `git log -1 --format=%s` check on
  each recorded SHA. This Change buys implementor **intent**, not
  audit-time enforcement.
- **Moving the slug definition into `CODING_STANDARDS.md`.** Slug
  generation is task-title slugification, owned by the implementor.
  `{slug}` stays as the implementor-supplied placeholder in the
  injected format.
- **Future format tokens** beyond `{change_name}`, `{task_id}`,
  `{slug}`. If new tokens are ever added (e.g. `{phase}`), the
  implementor's substitution step needs extending — out of scope.
- **Validator lint of `## Commits` body.** The skeleton written by
  `ai-harness init` ships `## Commits` as an empty heading; downstream
  repos fill it themselves. We rely on the orchestrator's missing-
  section detection to surface a malformed body; we do not add a
  schema lint in this Change.
- **Sentinel markers / drift timestamps (exploration Option C).**
  Overkill for v1; revisit if drift becomes a recurring audit issue.

## Capabilities

- **resolve-commit-format-from-standards**: At delegation time the
  orchestrator reads `CODING_STANDARDS.md` from the repo root,
  locates the `## Commits` heading, takes the first non-comment,
  non-blank line in its body as the canonical format, and inlines
  it into the delegation block sent to the implementor. If the file
  is missing, the `## Commits` heading is absent, or the body is
  empty, the orchestrator fails the delegation with a structured
  `status: blocked` error that names the missing artifact.
- **apply-injected-commit-format**: The implementor's loop step 6
  reads the injected format string verbatim, substitutes
  `{change_name}`, `{task_id}`, and `{slug}` (using the
  implementor-generated slug from the task title), and passes the
  result as the `-m` argument to `git commit`. If no format was
  injected, the implementor fails loud with `status: blocked`.
- **fail-loud-on-malformed-standards**: Both the orchestrator (file
  read / heading parse) and the implementor (missing injection)
  surface a structured blocked result with the specific missing
  artifact named, so the human owner of `CODING_STANDARDS.md` can
  fix the exact problem.

## Approach

### Reading mechanism — Option B (orchestrator injects)

The orchestrator already builds a "Skills to load before work" block
per delegation (`change-orchestrator.md:531-543`) and already injects
the TDD skill to the implementor (`change-orchestrator.md:555-562`).
We reuse that injection point: at delegation-build time the
orchestrator reads `CODING_STANDARDS.md`, resolves the format from
the `## Commits` body, and inlines it as a labeled directive in the
same block.

Tradeoffs accepted:
- **+** No per-task file IO in the implementor (format is read once
  per delegation in the orchestrator).
- **+** Smaller implementor prompt delta (~5 LOC): "use the
  `commit-format` directive inlined above; substitute
  `{change_name}`, `{task_id}`, `{slug}`."
- **+** Centralized parse-and-fail in the orchestrator matches the
  existing Blocking rule semantics.
- **+** Drift-free across delegations within a session because the
  orchestrator re-reads per delegation (current pattern).
- **−** The implementor trusts the orchestrator's parse — a
  malformed format the orchestrator missed would still produce a bad
  commit. Mitigated by the implementor's own loud-fail when no
  format is injected, and by Option A as a documented fallback path
  if Option B proves fragile.
- **−** Doubles one piece of the orchestrator↔implementor contract
  surface (the format string + the three substitution tokens).
  Acceptable: the tokens are stable and documented.

### Validator coverage

Out of scope. Documented as a follow-up Change: add a `git log -1
--format=%s` regex check on each SHA recorded in
`implementation.md ## Commits` against the format that was current
at delegation time. This Change buys **intent**; the follow-up buys
**enforcement**.

### File-missing fallback

**Fail loud.** Both layers must surface `status: blocked` with a
specific error:

- Orchestrator cannot read `CODING_STANDARDS.md` → blocks the
  delegation. Error names the missing file.
- Orchestrator cannot find `## Commits` heading → blocks. Error names
  the missing heading.
- Orchestrator finds heading with empty body → blocks. Error names
  the empty body.
- Implementor receives no injected format (defensive — should be
  unreachable given the orchestrator gates first) → blocks. Error
  names the missing directive.

Rationale: a silent fallback to a legacy format would hide
corruption. Loud-fail surfaces the problem at the first task, not
during a future audit. The existing Blocking rule at
`change-implementor.md:174-177` already supports this envelope.

### Format-string syntax

**Keep literal.** The format stays a single backtick-quoted line
under `## Commits`, e.g. `` `[{change_name}][{task_id}] {slug}` ``.
No richer grammar in this Change. Rationale: matches the current
contract; the validator's gates section also stores each gate as a
single literal markdown line — consistency.

### Multiple-format support

**Take the first non-comment, non-blank line under `## Commits` as
the canonical format.** If the section later grows multiple
formats, this Change picks the first one and ignores the rest. A
follow-up Change can introduce a richer selector if needed.

## Risks

- **R1 — Format-string typos.** A typo in `## Commits` body
  (`{change}` vs `{change_name}`) silently produces wrong commits.
  Validator does not check `git log` today, so the typo would not
  surface until archive time (or never).
  **Mitigation:** the orchestrator's parse-and-fail path catches
  the case where no format can be resolved. Token-level typos (wrong
  placeholder names) remain undetected at audit; surfaced as a
  follow-up Change that adds a low-cost regex lint.
- **R2 — Prompt bloat.** Adding a "use the injected commit-format
  directive" sentence to the implementor risks drift from the
  minimalist skill-loading posture that
  `change-validator.md:180-182` calls out. **Mitigation:** keep the
  directive short and label it as data, not skill. Mirrors the
  validator's "read the standards file literally, do not hardcode"
  phrasing.
- **R3 — Partial reads.** If a downstream repo edits
  `CODING_STANDARDS.md` mid-session, the orchestrator will pick up
  the new format on the next delegation (because it re-reads per
  delegation, per current pattern). Acceptable; matches today's
  "drift-free across delegations within a session" guarantee.
- **R4 — Slug corruption.** `{slug}` is implementor-generated. A
  malformed slug (whitespace, unicode, length > 72) produces a
  malformed commit subject. **Mitigation:** slug generation stays
  in the implementor (out of scope here). Acceptable; this Change
  does not alter slug rules.
- **R5 — Renderer indirection.** Implementor + orchestrator prompts
  are bundled resources rendered to per-CLI paths at install time
  (`src/ai_harness/modules/harness/renderers.py`). The edit lives in
  the source; render tests pick up the new content via the existing
  fixture pattern. **Mitigation:** add a dedicated renderer test
  asserting the new directive is present in both OpenCode and
  Claude outputs.
- **R6 — Intent vs enforcement gap.** This Change buys implementor
  intent only. A non-compliant commit will not fail audit until the
  follow-up `change-validator` Change lands. **Mitigation:**
  documented as explicit non-goal; the follow-up Change is the
  natural next step.
- **R7 — Archiver out-of-scope carve-out.** The archiver's
  `docs: archive {change}` (referenced in two places) stays
  hardcoded. If a future Change makes the archiver format
  configurable, both prompts move together. **Mitigation:**
  documented explicitly in this PRD; no drift risk for this Change.

## Rollback Plan

Revert the three source edits (implementor prompt, orchestrator
prompt, tests). Renderers will regenerate the per-CLI prompts on
next install. No data migration, no state on disk. Commit history
created under the new format stays as-is (humans and other tools
can still read the messages; only the implementor's future commits
revert to the old "include task id and change name" phrasing).

## Success Criteria

- **AC1.** Changing the body line under `## Commits` in
  `CODING_STANDARDS.md` from `` `[{change_name}][{task_id}] {slug}` ``
  to `` `feat({change_name}): {slug} [{task_id}]` `` and then
  completing a task results in `git log -1 --format=%s` showing the
  new format — **without editing any agent prompt file**.
- **AC2.** The implementor's per-task commit subject still includes
  the change name, task id, and a slug (order matches the format
  string).
- **AC3.** Deleting `CODING_STANDARDS.md` from the repo root causes
  the orchestrator to return `status: blocked` naming the missing
  file, **before** the implementor is spawned.
- **AC4.** Removing the `## Commits` heading from
  `CODING_STANDARDS.md` causes the orchestrator to return
  `status: blocked` naming the missing heading.
- **AC5.** Emptying the body under `## Commits` causes the
  orchestrator to return `status: blocked` naming the empty body.
- **AC6.** Renderers still produce equivalent OpenCode and Claude
  prompts; renderer tests cover the new directive in both targets.
- **AC7.** `e2e/e2e_test.sh` Tier-1 greps pass against the rendered
  implementor + orchestrator files in the home install dir.
